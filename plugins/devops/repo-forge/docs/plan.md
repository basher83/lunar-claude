## Rethinking Scope

Original plan was 5 separate skills: mise, pre-commit, gitignore, git-cliff,
project-setup. After reviewing config-aggregation.md and mise's preset/hook
capabilities, the architecture simplifies.

The real delivery mechanism is mise presets — executable task scripts that
scaffold repos using `mise use` (real version resolution), `mise tasks add`,
and `mise config set`. The skills teach Claude how to use and compose these
presets. The presets do the actual work.

This could collapse to:

1. **mise skill** (expanded) — covers config, tasks, hooks, presets, `mise
   generate`, and the version resolution pattern. The foundational knowledge
   layer.

2. **repo-onboard command or skill** — the "onboard this repo" directive that
   identifies project type and runs the right preset chain. Consumes everything
   the mise skill teaches.

The other originally planned skills (pre-commit, gitignore, git-cliff) become
reference material inside the mise skill or preset logic rather than standalone
skills. Their configs are generated by presets, not hand-written with skill
guidance.

## Key Design Constraints

- NEVER guess version numbers. Use `mise use <tool>` (pins current version at
  scaffold time by querying the registry), `mise ls-remote <tool>` (list
  available), or `mise ls` (what's installed). Training data versions are stale.
- Presets use `mise use` which resolves and pins at scaffold time.
- Mise hooks (enter, postinstall, watch_files) enable runtime validation,
  not just one-shot scaffolding.

## Research Findings (DeepWiki + Context7)

### `mise generate` — built-in scaffolding

Mise already has generation capabilities we should leverage instead of
reimplementing:

- `mise generate bootstrap` — creates a script that downloads and installs
  mise for contributors who don't have it. Supports `--localize` to sandbox
  mise into .mise/ within the project. Pair with `mise generate task-stubs`
  to let contributors run tasks without global mise.

- `mise generate config` — creates a mise.toml, can import from existing
  .tool-versions files.

- `mise generate git-pre-commit` — generates a git pre-commit hook that runs
  a mise task (e.g., `mise run pre-commit`). Complements the pre-commit
  framework rather than replacing it. Passes staged files via $STAGED.

- `mise generate github-action` — generates a CI workflow that runs `mise run
  ci` on push. Uses jdx/mise-action.

- `mise generate task-docs` — generates documentation for all defined tasks.
  Can inject into existing files or render per-task docs.

- `mise generate task-stubs` — creates shims in bin/ so contributors can run
  `./bin/<task>` without mise installed. Pairs with bootstrap.

Key insight: presets should call `mise generate` where appropriate rather than
writing files from scratch. For example, preset:base should run
`mise generate bootstrap --localize --write` and
`mise generate git-pre-commit --task=pre-commit --write`.

### `mise use` — version resolution lifecycle

1. Parses TOOL@VERSION argument
2. Selects config file (local mise.toml, global, or --path target)
3. Resolves version: checks lockfile first, then queries backend registry
4. Installs if not present
5. Writes pinned version to mise.toml
6. Rebuilds shims

Default is fuzzy pinning (`mise use python@3.13` writes `python = "3.13"`).
Use `--pin` for exact (`python = "3.13.2"`). This is the correct mechanism
for presets — never hardcode versions.

### `mise config set` — programmatic configuration

Can set arbitrary config paths:
- `mise config set tools.python 3.12`
- `mise config set env._.python.venv.path .venv`
- `mise config set env._.python.venv.create true -t bool`
- `mise config set settings.jobs 4`
- Supports --type flag: string, bool, list, set

### Registry and backends

Mise has a three-layer architecture:
1. Registry layer — maps short names to backends (compiled from TOML files)
2. Backend resolution — converts to concrete installers
3. Installation — download, verify, extract, install

Backends: core (node, python, ruby, go, java), aqua (ideal for new tools,
no plugins needed, works on Windows), cargo, npm, pipx, go, github, gitlab,
asdf (legacy), vfox, dotnet, gem, conda, http, s3.

The aqua registry is compiled into the mise binary — no external lookups
needed for tools registered there.

### Task dependency system

Three dependency types:
- `depends` — must complete before task runs
- `wait_for` — soft dependency, waits only if also being run
- `depends_post` — runs after task completes (success or failure)

Dependency args and env can be passed inline:
```toml
[tasks.deploy]
depends = [
  { task = "build", args = ["--release"],
    env = { RUSTFLAGS = "-C opt-level=3" } }
]
```

Tasks support `sources` and `outputs` for incremental execution — skip if
outputs are newer than sources. Useful for build/compile tasks.

### Hooks

Events: enter (once on project entry), cd (every dir change), leave (exit
project), preinstall, postinstall, watch_files.

Can trigger mise tasks directly:
```toml
[hooks]
enter = { task = "setup" }
```

Shell hooks execute in current shell (can source files):
```toml
[hooks.enter]
shell = "bash"
script = "source completions.sh"
```

watch_files monitors patterns and auto-runs:
```toml
[[watch_files]]
patterns = ["src/**/*.rs"]
run = "cargo fmt"
```

postinstall gets $MISE_INSTALLED_TOOLS (JSON array) and per-tool
$MISE_TOOL_NAME, $MISE_TOOL_VERSION, $MISE_TOOL_INSTALL_PATH.

### `hk` — pre-commit replacement (jdx/hk)

hk is a Rust-based git hook manager by the same author as mise. It replaces
the pre-commit framework entirely with significant advantages:

Architecture:
- Read/write file locks enable aggressive parallelism without race conditions
- `check` steps get read locks (run concurrently), `fix` steps get write locks
- `check_first = true` (default): runs check with read lock first, only
  acquires write lock if check fails. Optimizes the common case.
- Uses libgit2 directly instead of shelling out to git (faster)
- Stashes unstaged changes before fix hooks (solves the pre-commit staged/
  unstaged mixing bug)

Configuration: hk.pkl (Apple's PKL language)
- Typed configuration with schema validation
- Builtins for common tools (no version pinning of hook repos needed)
- Also supports TOML/YAML/JSON but builtins only available in PKL

Builtins (built-in linter/formatter definitions):
- JS/TS: prettier, eslint, tsc, biome, deno
- Python: black, ruff, mypy, pylint
- Rust: cargo_fmt, cargo_clippy, rustfmt
- Go: go_fmt, go_imports, golangci_lint, go_sec
- Ruby: rubocop, standard_rb
- Shell: shellcheck, shfmt
- Config: yamllint, yamlfmt, taplo (TOML), jq (JSON), pkl
- Markup: markdown_lint, lychee (link checking)
- Security: detect_private_key
- Git: check_merge_conflict, check_case_conflict, no_commit_to_branch
- Files: trailing_whitespace, newlines, mixed_line_ending, fix_smart_quotes
- Meta: check_conventional_commit, mise (config validation), tombi

Mise integration:
- `hk init --mise` — generates hk.pkl AND mise.toml with hk configured
- `hk install --mise` — hooks use `mise x` for execution
- `HK_MISE=1` env var enables mise integration by default
- Tools managed via mise, hooks managed via hk — clean separation

Migration from pre-commit:
- `hk migrate pre-commit` — reads .pre-commit-config.yaml, generates hk.pkl,
  converts repos to builtins where possible. Preserves hooks, patterns,
  and excludes.

Profiles:
- Steps can have `profiles = List("slow")` to only run in specific contexts
- `HK_PROFILE=slow hk check` or `hk check --slow`
- Useful for separating fast pre-commit checks from slow CI checks

Hook types: pre-commit, pre-push, prepare-commit-msg, commit-msg
Plus `hk check` and `hk fix` as standalone commands (not tied to git hooks).

Impact on repo-forge:
This eliminates the entire pre-commit version drift problem from the
aggregation report. No more .pre-commit-config.yaml with stale hook repo
versions. Builtins resolve tool versions through mise. The migration path
exists (`hk migrate pre-commit`) for existing repos.

## Migration Path for Existing Repos

For the 19 repos currently using pre-commit:
1. `hk migrate pre-commit` — auto-converts .pre-commit-config.yaml to hk.pkl
2. Review generated hk.pkl, replace custom hooks with builtins where possible
3. `hk install --mise` — set up git hooks
4. Remove .pre-commit-config.yaml
5. `mise rm pre-commit` — remove pre-commit from mise tools

This can be a repo-onboard subcommand or a separate migration command.

## Resolved Decisions

- hk replaces pre-commit. Confirmed. 36 releases in 13 months, zero open
  bugs, same author as mise, built-in migration from pre-commit, builtins
  cover the full stack. hclfmt builtin added v1.36.0 for Terraform.
- PKL is the config format for hk. Builtins require it. The `hk init`
  auto-detection (v1.35.0) minimizes the PKL learning curve — most config
  is generated, not hand-written.
- Presets live in ~/.config/mise/tasks/preset/ (global). They scaffold new
  repos that don't have the plugin yet. The mise skill in repo-forge teaches
  how to build, use, and customize presets.
- Version resolution: always `mise use <tool>` for pinning. Never hardcode.
  Default fuzzy pin (e.g., `python = "3.13"`), use `--pin` only when patch
  version matters.

## Unresolved Decisions

- Line length: 88 vs 100 vs 120. Needs a call. 100 is the compromise.
- Ruff rule baseline: E, F, I, UP minimum. B, SIM, PTH for new projects.
- repo-onboard: skill (auto-activates) vs command (explicit /repo-onboard).
  Leaning command — you want to control when scaffolding happens.
- `mise generate bootstrap --localize` vs non-localized as default.
